## Purpose
Define the initial steps to validate and authorize work, align stakeholders, and create a lightweight plan.

## When to use
Whenever a new project idea or feature proposal is ready to be explored.

## Goals
- Confirm business need and measurable outcome
- Identify stakeholders & champions
- Define success criteria and initial timeline
- Decide go/no-go for planning

## Minimum Deliverables
- Project One-pager (Problem, Goal, Success Metrics)
- Stakeholder list & communication plan
- High-level timeline and key milestones
- Initial risk list
- Resource needs (team roles, rough effort estimate)

- Project name:

- Objective / Goal (SMART):
- Success metrics:
- Primary stakeholders:
- Suggested timeline / milestones:
- Quick risks & dependencies:
- Proposed team / roles:

## Initiation Checklist
- [ ] One-pager completed and reviewed by Product Lead
- [ ] Sponsor / Stakeholder alignment (email or meeting)
- [ ] Stakeholder analysis completed (see template below)
- [ ] Decision: Approve to move into planning?
- [ ] Create repo or project board skeleton
- [ ] Add initial artifacts to repo (docs/ or .copilot/)
- [ ] Team onboarding scheduled (see onboarding checklist below)

## Stakeholder Analysis Template

| Stakeholder | Role | Interest/Impact | Communication Frequency | Primary Contact |
|-------------|------|-----------------|------------------------|-----------------|
| [Name] | [Role] | [High/Med/Low Impact] | [Daily/Weekly/Monthly] | [PM/PdM/CSM] |

**Influence/Interest Matrix:**
- High Influence, High Interest: Manage closely (e.g., Sponsor, Product Lead)
- High Influence, Low Interest: Keep satisfied (e.g., Executive stakeholders)
- Low Influence, High Interest: Keep informed (e.g., End users, CSM)
- Low Influence, Low Interest: Monitor (e.g., Indirect teams)

## Team Onboarding Checklist

### For Project Managers
- [ ] Review project one-pager and success metrics
- [ ] Set up project board and tracking tools
- [ ] Schedule kickoff and recurring meetings
- [ ] Create initial risk register
- [ ] Identify cross-team dependencies

### For Developers and QA
- [ ] Access to repository and development environment
- [ ] Review technical architecture and design docs
- [ ] Understand definition of done and testing requirements
- [ ] Set up CI/CD pipeline access
- [ ] Review coding standards and PR guidelines

### For Product/Business Analysts
- [ ] Review product vision and roadmap
- [ ] Access to requirements and specifications
- [ ] Understand customer segments and personas
- [ ] Review acceptance criteria templates
- [ ] Set up analytics and metrics dashboards

### For UX Designers
- [ ] Review brand guidelines and design system
- [ ] Access to design tools and assets
- [ ] Understand user research findings and personas
- [ ] Review accessibility requirements
- [ ] Set up design review cadence

### For Security Leads
- [ ] Review security policies and compliance requirements
- [ ] Access to security scanning tools
- [ ] Understand data classification and handling requirements
- [ ] Set up security review checkpoints
- [ ] Review incident response procedures

### For Customer Success Managers
- [ ] Understand customer base and segments
- [ ] Access to customer feedback channels
- [ ] Review customer health metrics
- [ ] Understand escalation procedures
- [ ] Set up customer communication plan

### For Scrum Masters
- [ ] Establish team working agreements
- [ ] Set up agile ceremonies and cadence
- [ ] Identify team impediments and escalation paths
- [ ] Review velocity and sprint planning approach
- [ ] Set up retrospective format and tracking

## Decision Gate
Move to planning when:
- Success metrics are clear
- Stakeholders agree on priority
- Team availability is confirmed
